The science of crawl (1): deduplication of web content

Here at URX we are building the world's first mobile app search API. Backing this API is a search engine containing a large corpus of web documents, meticulously maintained and carefully grown by crawling internet content. We've come to discover that building a functional crawler can be done relatively cheaply, but building a robust crawler requires overcoming a few technical challenges. Christopher Olston and Marc Najork give a fantastic overview of these challenges; many of our practices are inspired by Olston and his referenced papers.

In this first installment, I will walk through the duplicate web content problem. When crawling the web using a standard link follower (our favorite being Scrapy) it's fairly common to run across the same link more than once. If this link is crawled more than once, a duplicate of this page will be added into our search index. Unless the content has materially changed since the last crawl (topic for another blog post), we will begin to build multiple copies of the same web page in our search index.

Duplicate pages in a search index poison search results. The goal of a search engine is to return both relevant and diverse documents, allowing users to decide the optimal resolution for their own query. Without deduplication, the top-k results returned for a user's query may be k copies of the same page. This creates a bad user experience. In fact, as the crawler scales out, the likelihood of observing such duplicate poisoning increases. In fact, Google's Matt Cutts believes that up to 20% of web content is duplicated.

Looking beyond exact duplicate links, a given piece of content can be duplicated across different domains. Now, we have to watch for the same content appearing anywhere on the web. This encourages a solution which looks beyond the URL and assesses similarity among the HTML page. Many pages across domains share similar content - with only an ad counter or a html template differing between pages. We now must look for a solution which measures similarity among web documents. This last problem is referred to as near-duplicate detection.

Fortunately, the art of detecting duplicate and near-duplicate content is a well researched. We present our deduplication process as a funnel. At the top we aim to detect a large percentage of obvious, exact duplicate content with minimal memory overhead. As you move down the funnel techniques for near-duplicate detection become more robust to small differences in page content at the expense of larger memory requirements. Let's tackle this one step at a time.

Duplicated URLs

The goal at the top of the deduplication funnel is to discover and remove a large number of obvious duplicate URLs. To this end a hash map of previously crawled URLs can be maintained. Unfortunately the memory requirements of such a map will grow large as the number of unique URLs we crawl approaches web scale. To avoid a memory explosion but retain O(1) duplicate lookups, a bloom filter is used. Bloom filters are beautiful, constant-space, collision data structures with guaranteed perfect recall at the cost of imperfect precision. This means if a URL has been added into the filter it will always be detected as a duplicate. However, a number of new URLs will be falsely declared duplicate. The precision of a bloom filter can be tuned at the expense of additional (constant) memory and compute time. We prefer to err on the side of falsely dropping a small amount of novel content in exchange for removing a majority of duplicate pages.

Fortunately for us, Scrapy maintains a built in duplicate URL server which avoids re-crawling the same URLs.

Textually similar URLs

Often URLs for a given page are not exactly the same. There are parameters embedded within a url which may differ. For example, both of these pages, P1 and P2 link to the same Ticketmaster location page even though the URLs have different URI parameters. To this end, Ziv Bar-Yossef, et al. wrote about identifying DUST: different URLs with similar text. Their method mines crawl logs to learn canonical representations for a given site and builds a set of substring and parameter substitution rules to canonicalize uncrawled links. Given a large enough web corpus, this method can clean the above links into P1* and P2*. After URLs are canonicalized, the above bloomfilter can be used to deduplicate URLs.

Duplicated content

The above methods work well for identifying duplicate content within a given pay level domain (e.g., ticketmaster.com). Looking for duplicate content across domains requires crawling a link and retrieving the corresponding HTML page. Similar to duplicated URLs, once a page is returned to the crawler, duplicate content can be efficiently detected with (you guessed it!) another bloom filter. In this case we can no longer rely in Scrapy's implementation (built for link resolution) and we default to using an efficient python implementation.

from pybloomfilter import BloomFilter

false_positive_probability = 0.0001

print 'is orange in the bf? %s' % ('orange' in bf)

Similar to canonicalizing links, the next step in the funnel is to clean HTML content. Two pages may share the same visible content while having subtly different HTML elements. There are several methods for removing HTML text - an efficient method crawls LXML etree's and looks for html node text. A simpler to implement version leverages NLTK's clean_html:

In [14]: 
from nltk import clean_html

# Two pages, textually equivalent, with different HTML tags
page1 = '<html><body><b>hi!</b></body></html>'
page2 = '<html><body><i>hi!</i></body></html>'

# check for duplicate content
bf = BloomFilter(1E6, .0001)
for page in (page1, page2):
    if bf.add(page):
        print 'Duplicate "%s" Detected' % page

clean_page1 = clean_html(page1)
clean_page2 = clean_html(page2)

# check again for duplicate content
for page in (clean_page1, clean_page2):
    if bf.add(page):
        print 'Duplicate "%s" Detected' % page

Duplicate "hi!" Detected

Textually similar content: Locality Sensitive Hash (LSH)

The above methods work well in identifying a majority of exact duplicate content. However, there remains a class of near-duplicate web content where visible-text elements may be slightly different that can still poison search results. To identify such content, we move to a more computationally intensive method for identification: locality sensitive hashing. A great overview of LSH was published a few days ago on Eventbrite's blog; we present a similar view.

Let's start with a simple example. Two strings "My beautiful dog, Woof" and "My beautiful dog, Woo" are similar but not the exact same. We would like to detect such cases as they are semantically equivalent. Conventional string similarity methods require either:

  1. Exact string matches (discussed above) or
  2. Computationally intensive methods

Let's look at an example of 1.

In [16]: 
s1 = "My beautiful dog, Woof"
s2 = "My beautiful dog, Woo"

s1 == s2
Out[16]:
False

Unsurprisingly, the two strings are not literal equivalents. String comparisons are blazingly fast in python, through a combination of hashtables and interning but useless for the non-literal problem.

A common pairwise, string similarity method is to calculate the Levenshtein Distance. This metric calculates the edit distance between two strings, i.e., the number of character edits (insertion, deletion or mutation) to change one string into a second string. For example, in the case of "cat" and "car" the edit distance is 1. The distance between s1 and s2 above is also one. A naive implementation yields a \(O(2^N)\) algorithm where N is the length of the longer input string. There are better \(O(N*M)\) and even \(\Theta(n^{1-\Theta(1/log\ log\ n)})\) algorithms for Levenshtein distance calcuations, but they are all too expensive to use in general.

We require a fast method to 1. compare two text documents and calculate a similarity metric and 2. calculate this metric for a new document against a large corpus of documents. For this reason, we turn to locality sensitive hashing. Let's take a look at how it works. Take one of our strings above. First, clean the string by removing punctuation, capitals and spaces

Measuring feature similarity as a proxy to text similarity is common in language processing. One of the most well known methods is the Jaccard index. Mathematically, the Jaccard distance, \(J\), between feature vectors \(A\) and \(B\) is given as

\[J(A, B) = \frac{\left | A \cap B\right | }{\left | A \cup B\right | } \]

This simple measure calcuates the number of features \(A\) and \(B\) share and normalize it by the total number of distinct features present in \(A\) and \(B\). Lets take our two feature vectors that we extracted from our two above strings.

In [32]: 
def Jaccard(A, B):
    return (len(set(A) & set(B))) / float(len(set(A) | set(B)))

# Jaccard of the two feature vectors above
print Jaccard(feature_vector1, feature_vector2)

# Jaccard of two *very* different strings
print Jaccard(["123", "234", "345"], ["345", "456", "567"])

0.9375
0.2

The Jaccard successfully measures string similarity by mapping feature vectors to a float between 0 and 1. Unfortunately this method is difficult to scale for solving 2.: comparing one document against a large corpus of documents. To this end we turn to a probabilitic string encoding: LSH.

For LSH, a set of hash tables is generated, each representing a band or bucket. Every feature extracted from a page is hashed into its own band. Ultimately the similarity metric will be derived from a weighted sum of the number of overlapping bands. This acts as a proxy to the number of overlapping features (shingles). The number of bands used presents a tradeoff between the speed of evaluating the LSH and the accuracy of identifying near-duplicate content.

The neat thing here is that, in each band the probability that two similar hashes collide is higher if the input features are similar. If we have two strings where some of the tokens hash to the same value, then we expect more collisions than from two randomly generated strings.

Putting it together as a SimHash¶

One well known implementation of LSH is Google's Simhash. This publication was released alongside Google's description of implementing Simhash at "web scale". We summarize the key points below.

Each feature extracted from a text document is hashed, \(h_i\) and placed into a component of a large hash vector, \(H\):

\[ H = \begin{pmatrix}h_1\\h_2\\...\\h_n\end{pmatrix} \]

\(h_i\) is typically a 64-bit hash (jenkins, murmur3, cityhash, etc.) of n-mer \(i\) in S. Each \(h_i\) is bitwise and of 64-bit binary masks, \(m_j\), where

\[ m_j = \begin{cases} 1 & bit\ j\\ 0 & else \end{cases} \]

From this, a feature-hash is calculated using the following mapping

\[ V_i = SIGN\left(\sum_{j = 1}^{64}{h_i \& m_j}\right) \]

where the operator \(\&\) is defined as:

\[ h_i \& m_j = \begin{cases} 1 & if\ 1\\ -1 & if\ 0 \end{cases} \]

and where we calculate the \(SIGN\) as follows

\[ SIGN\left(x\right) = \begin{cases} 1 & if\ x>0\\ 0 & if\ x\leq0 \end{cases} \]

Using this construction, every n-mer in S contributes to each of 64 feature-hashes in \(V_i\). This creates a robust similarity metric where small changes in S will not result in large bit changes to \(V_i\). The formal process can be easier to comprehend in visual form:

In the end, this is the exact property we're looking for! A hash function which maps entities with small changes to the same (or nearly the same) bit vector. Let's try it out in practice

"""
Taken from:
https://github.com/liangsun/simhash/blob/master/simhash/__init__.py

Renamed Simhash to Bithash to avoid clash with modules Simhash-py and Simhash-db from seomoz
"""

#Created by Liang Sun in 2013
import re
import logging
import collections

class Bithash(object):
    def __init__(self, value, f=64, reg=ur'[\w\u4e00-\u9fff]+', hashfunc=None):
        '''
        `f` is the dimensions of fingerprints

        `reg` is meaningful only when `value` is basestring and describes
        what is considered to be a letter inside parsed string. Regexp
        object can also be specified (some attempt to handle any letters
        is to specify reg=re.compile(r'\w', re.UNICODE))

        `hashfunc` accepts a utf-8 encoded string and returns a unsigned
        integer in at least `f` bits.
        '''

        self.f = f
        self.reg = reg
        self.value = None

        if hashfunc is None:
            import hashlib
            self.hashfunc = lambda x: int(hashlib.md5(x).hexdigest(), 16)
        else:
            self.hashfunc = hashfunc

        if isinstance(value, Bithash):
            self.value = value.value
        elif isinstance(value, basestring):
            self.build_by_text(unicode(value))
        elif isinstance(value, collections.Iterable):
            self.build_by_features(value)
        elif isinstance(value, long):
            self.value = value
        else:
            raise Exception('Bad parameter')

    def _slide(self, content, width=3):
        return [content[i:i+width] for i in xrange(max(len(content)-width+1, 1))]

    def _tokenize(self, content):
        ans = []
        content = content.lower()
        content = ''.join(re.findall(self.reg, content))
        ans = self._slide(content)
        return ans

    def build_by_text(self, content):
        features = self._tokenize(content)
        self._features = features
        return self.build_by_features(features)

    def build_by_features(self, features):
        hashs = [self.hashfunc(w.encode('utf-8')) for w in features]
        v = [0]*self.f
        masks = [1 << i for i in xrange(self.f)]
        for h in hashs:
            for i in xrange(self.f):
                v[i] += 1 if h & masks[i] else -1
        ans = 0
        for i in xrange(self.f):
            if v[i] >= 0:
                ans |= masks[i]
        self.value = ans

    def distance(self, another):
        x = (self.value ^ another.value) & ((1 << self.f) - 1)
        ans = 0
        while x:
            ans += 1
            x &= x-1
        return ans

It's great that we have a robust way of calculating page similarity, but the whole reason for avoiding the Jaccard measure is avoid costly all-pairwise corpus comparisons. While full details are beyond the scope of this post, there is a really neat method described by Google to solve this exact problem. Roughly, several copies of the simhashed corpus are maintained. Each copy sorts the index by a different rage over the bits. It then becomes trivial to discover simhashes differing from an input hash by at most k-bits (i.e., identifying near duplicates) via a constant number of divide-and-conquer searches. This amortizes to a logarithmic look-up operation.

In the end

There is no one, perfect method for identifying duplicate web content. The web itself is too large and too complex. We can, however, devise a funnel of methods which, when chained together, provide "acceptable" duplicate detection accuracy. The process is relatively fast - a series of bloom filters, string cleaners, bit hashes and, finally, a logarithmic time lookup. This funnel does not catch every duplicate page, but, on average, it cleans our corpus of web documents and ultimately provides more diverse search results to users.
